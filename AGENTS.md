# AGENTS.md

## 工作约定

- 每次执行开发任务前，先阅读 `AGENTS.md` 和 `docs/GDD.md`。
- 当前项目使用 Godot 4.x + C#，不要改成 GDScript。
- 不要引入不必要的复杂架构；以 1 天内可演示 Demo 为目标。
- 每次只完成用户指定的当前 Step，不要提前实现后续功能。
- 不要删除已有功能，不要大规模重构。
- 不要提交或推送 Git，除非用户明确要求。
- 修改 JavaScript 文件后必须运行 `npm test`。
- 安装依赖优先使用 `pnpm`。
- 添加新的生产依赖前必须先询问用户确认。

## 项目目标

本项目是《无限列车 SLG》的 Godot 4 C# 灰盒 Demo。目标不是完整商业 SLG，而是验证一个最小但完整的前期循环：

```text
列车主界面 -> 前往废弃站点 -> 竖屏双轨道防守战斗 -> 击败 Boss
-> 获得资源 -> 返回列车 -> 升级炮台 / 列车 -> 下一局更强
```

玩家经营的是一列末日列车，而不是固定城市。

## 设计优先级

- 以 `docs/GDD.md` 为最高设计依据。
- 如果旧代码、README 或其它文档与 `docs/GDD.md` 冲突，以 `docs/GDD.md` 为准。
- 如果发现设计冲突，不要擅自扩展系统；优先采用最小可玩方案，并在回复中说明取舍。
- 当前阶段优先保证 Demo 可运行、可验收、可快速迭代。

## 分辨率和坐标规则

- 内部玩法分辨率固定为 `540 x 960`。
- 画面比例为 `9:16` 竖屏。
- 桌面窗口可以放大显示，但不要为了桌面显示修改玩法坐标。
- 推荐区域：

```text
0 - 160       Boss / 远景威胁区
160 - 700     敌人、升级目标、子弹飞行区
700 - 820     英雄判定线 / 玩家操作核心区
820 - 960     横向列车基地 / 炮台安装区 / 底部 UI 安全区
```

- 推荐轨道位置：

```text
左轨 x = 175
右轨 x = 365
```

## 战斗核心规则

- 战斗必须是竖屏双轨道防守玩法。
- 底部是固定不动的横向列车基地。
- 敌人和升级目标必须从屏幕上方出现，并向下压向底部列车。
- 不要把战斗做成“敌人从右侧出现并向左靠近列车”。
- 玩家通过左右拖动控制英雄和炮台的横向位置。
- 英雄和炮台共享同一个 x 坐标。
- 英雄和炮台自动向上射击，玩家不需要点击开火。
- 炮台必须看起来安装在列车上，不要悬浮。

## 列车主界面规则

列车主界面必须显示：

- Scrap
- Fuel
- Food
- Parts
- TrainLevel
- CannonLevel
- CurrentStation
- Train HP
- 升级炮台按钮
- 前往废弃站点按钮

Demo 阶段必须实现炮台升级，并让下一次战斗火力变强。列车升级和修理列车可以后续补，但不要做复杂科技树、建筑队列、装备、抽卡、联盟、国战或真实 SLG 大地图。

## 推荐开发步骤

按顺序开发，不要跳步：

```text
Step 1：项目分辨率和基础场景
Step 2：GameManager 和 TrainScreen 资源显示
Step 3：BattleScreen 画面布局和拖动操作
Step 4：自动射击和子弹
Step 5：基础敌人
Step 6：可射击升级目标
Step 7：普通波次节奏
Step 8：Boss 和轨道预警
Step 9：结算、返回列车、发奖励
Step 10：反馈、UI、调参
```

每完成一步后，必须说明：

- 修改了哪些文件。
- 实现了哪些功能。
- 如何在 Godot 中测试。
- 是否有未完成项或风险。
